# Контент-фабрика (`/content-article`)

Оркестрирую «редакцию» из готовых агентов. Сам тяжёлую работу не делаю — делегирую,
чтобы не раздувать главный контекст (токен-экономика). Веду фаундера по шагам.

## Шаг 0 — собрать вводные
Спроси (если не задано): **тема**, **целевой рынок/страна** (определяет локаль и
источники данных), **проект**, **формат** (1 статья или серия).
Локаль контента = страна кампании; дефолт `uk`/`en`.

## Конвейер
1. **План** (если просят серию/кластер) → агент `pm-content-strategist`
   (+ при желании **агент** `seo-cluster` для hub-and-spoke — спавни как субагента,
   скилл скрыт). Согласуй темы с фаундером.
2. **Бриф** на конкретную статью → скилл `/seo-content-brief` (ключ, структура, что осветить).
3. **Данные** → агент `content-researcher`: тянет уникальные цифры из официальных
   источников по стране (реестр источников данных) →
   факт-лист со ссылками. (Нет данных → честно помечает, не выдумывает.)
4. **Письмо** → агент `content-writer-longform`: пишет уникальный лонгрид (бриф + факт-лист
   + бренд-голос + SEO/GEO-правила) В СВОЁМ окне.
5. **QA** → агент `seo-content`: E-E-A-T, читаемость, AI-citation, тонкий контент.
6. **AI-search** → скилл `/ai-search-ready`: FAQ, FAQPage schema, answer-block, anchor.
7. **Публикация** — по правилам проекта (в БД/блог; деплой через ваш deploy-флоу).
   Сам НЕ публикую без подтверждения фаундера.

## Параллельность
Для серии статей: шаг 3-4 можно гонять параллельно по темам (каждая в своём окне агента).

## Что НЕ делать
- Не выдумывать цифры/источники (это правило агентов — следи за ним).
- Не публиковать без «ок» фаундера.
- Не ставить python-пакеты без `oss-vetter` (по умолчанию данные тянем через curl/WebFetch).
- Не смешивать бренд-голоса разных проектов.

## Связи
Агенты (спавнить как субагентов): `content-researcher`, `content-writer-longform`,
`pm-content-strategist`, `seo-content`, `seo-cluster`.
Скиллы (вызываются): `/seo-content-brief`, `/ai-search-ready`, ваш deploy-скилл (публикация).
Знание: реестр источников данных для контента.
